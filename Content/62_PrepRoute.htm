Preparation for Routing - Mesh and Ring

Preparation for routing entails creating network objects and entering values into object properties.  A minimum set of network information is required for the routing algorithms to function properly.  However, you can enter additional information in the form of objects and object properties, that will be factored into your routing results.

Minimum Routing Requirements

In preparation for routing demands, the following information must be present:

  • A set of locations
  • A set of lightpaths connecting the locations
  • A set of demands to be routed between locations
  • Values for the following lightpath properties:
    • Length - must be greater than zero
    • AdminCost - must be greater than zero
    • Type (i.e. OC192_STM64)

  • Values for the following demand properties:

    • Type (i.e. OC48_STS48_STM16)
    • Quantity
    • Protection Type (i.e. OnePlusOne)
    • Scope (Mesh or Ring)

     

Note: Fiber objects are not required for routing to occur.  However, if fiber objects are present, then the information they contain is used by the routing algorithms.

In addition to the above, networks with rings require the following:

  • One or more rings must be present.  (Note:  The Ring Cover algorithm is a quick way to create rings using existing location and fiber information).
  • At least one demand must have it's Scope property set to Ring.

  • Values for the following ring properties:

    • Type (i.e. OC192_STM64)
    • Architecture (i.e. BLSR2F_MSSPRING2F)

The majority of this information is often included in the network file used to initiate the design process.  Additional information can be added by using the Property editor to enter values. 

Note:  When fiber information is available, the amount of data that must be entered to create a network file can be minimized by deriving values for LightPath Length and LightPath AdminCost properties from Fiber Length.  Instructions on how to accomplish this are provided in the Derive LightPath Properties from Fiber Length section of this help file.  Fiber and LightPath Lengths can also be derived from location coordinates.  For descriptions of these commands, read the Derive Fiber Length From Great Circle Distance and Derive LightPath Length From Great Circle Distance sections of this help file.

Object Properties that Influence Routing Results

Some of the object properties that impact routing results are described in the table below.  Values highlighted in bold are required for the routing algorithms to generate valid routing results.   

Routing Related Object Properties

Object Property Description
Location DoNotUse If True, then the DoNotUse property of the lightpaths terminating  at this location will be set to True preventing them from being used for routing demands.
PermitThroughTraffic Use this property to specify the type of lightpaths that will be allowed to support through traffic (traffic that does not terminate at a location) at a given location. Valid entries include:
  • AnyLightPath
  • ExpressLightPaths
  • None
Type Location type.  Valid entries include: 
  • SN16000 
  • SN9000
  • SN3000
  • Switch
Fiber ConduitIDs The sequential list of conduit ids (colon delimited) assigned to this fiber.
DoNotUse When true, the lightpaths associated with this fiber are excluded from the routing process.  
Length The relative distance between the start and end locations of this fiber.  Units (kilometers or miles) are determined by the LengthUnit property of the Network object.  Values for fiber length can be derived from location coordinates by using the command described in the Derive Fiber Length From Great Circle Distance section of this help file.
LightPath CapacityMaximum Use this property to specify an upper limit target for lightpath capacity.  The routing algorithms use CapacityMaximum in conjunction with Routing Constraint settings to make routing decisions.
CapacityMinimum Use this property to specify an lower limit target for lightpath capacity.  The routing algorithms use CapacityMinimum in conjunction with Routing Constraint settings to make routing decisions.
AdminCost Use this property to assign a "cost" value to each lightpath.  When you set the AdminCosts for the lightpaths in your design, you are in effect assigning a relative priority to each lightpath (higher cost = lower priority).  These priorities are factored into the decisions made by Modeler's routing algorithms.  The Set LightPath Admin Costs tool allows you to set the admin costs of a group of lightpaths by applying a simple equation.
ConduitIDs The sequential list of conduit ids (colon delimited) assigned to this lightpath.
DoNotUse When True, the lightpath is excluded from the routing process.  
Length The relative distance between the start and end locations of this lightpath.  Units (kilometers or miles) are determined by the LengthUnit property of the Network object.  If your network file includes fiber length data, then you can automatically assign  lightpath length values by following the instructions in the Derive LightPath Length from Fiber Length section of this help file.
Type Lightpath type.  Valid entries include:
  • OC3_STM1
  • OC12_STM4
  • OC48_STM16
  • OC192_STM64
VPN The number of the virtual private network (VPN) this lightpath is assigned to.
Ring Architecture Ring architecture.  Valid entries include:
  • BLSR2F_MSSSPRING2F
  • BLSR4F_MSSSPRING4F
  • ASYM4F
  • UPSR_SNCP
Type Ring type.  Valid entries include:
  • OC3_STM1
  • OC12_STM4
  • OC48_STM16
  • OC192_STM64
Demand DoNotRoute When True, demand will be ignored by routing algorithms.
Quantity Demand size.  Units are based on the value of the Type property.
Scope Demand scope.  Valid entries include:
  • Mesh 
  • Ring 
Type Demand type.  Valid entries include: 
  • AU3, DS-3, VC3, VC4, E3
  • STM-1, STM-4, STM-16, STM-64
  • STS-1, STS-3, STS-12, STS-48, STS-192
  • OC-1, OC-3, OC-12, OC-48, OC-192
BroadleafProtection Type of Broadleaf Protection.  Valid entries include: 
  • Basic 
  • DiverselyRouted
  • Conduit_ID_Diverse
BroadleafReversion Type of Broadleaf Reversion.  Valid entries include: 
  • Disabled 
  • Manual
  • Automatic
BroadleafSourceReroute When On, Broadleaf Source Reroute is enabled.
BroadleafTrunkRestoration When On, Broadleaf Trunk Restoration is enabled.
NodeProtection When True, node protection is enabled.  
ProtectionType Demand Protection Type.  Valid entries include: 
  • Unprotected (working only) 
  • OnePlusOne (working and protect) 
  • SharedPath (working and protect) 
FixedLightPathRouting When True, the lightpath routing of a demand cannot be changed by a routing algorithm.
FixedLightPathTimeSlots When True, the timeslot assignments of a demand cannot be changed by a routing algorithm.  When True, FixedLIghtPathRouting must also be True.
VPN The number of the virtual private network (VPN) this demand is assigned to.
