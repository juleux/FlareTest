Demand Routing - Mesh

The Route Demands algorithm combines the routing of work and protect paths for all demand protection types (UnProtected, OnePlusOne and SharedPath).

You can run the Route Demands algorithm using either default or advanced routing options.  Default routing options are in effect from the moment you install the Modeler.  Their values are carefully selected to provide optimum routing results for a majority of applications.  Routing options retain their default values unless you elect to edit them using either the Advanced tab or the Routing Constraints dialog.     

Note:  Once edited, default routing option values can be reinstated.  The majority of routing options return to their default values upon closure of the Route Demands dialog, others are reset when the network file is closed, and still others are reset only when you click a button to return them to their default value.  For instructions on reinstating the default values of routing options that have been edited, refer to the Apply Default Routing Options section of this help file.

Advanced routing options are helpful in those rare situations where default parameters do not provide the desired results.  Each routing option is described in the Advanced Routing Options section of this help file.

Once you have determined which options to use, follow the steps below to initiate the routing process:

  1. Create any additional objects or object properties necessary to define your network.  Refer to the list of Minimum Routing Requirements described in the Preparation for Routing section of this help file.
  2. Select Algorithms, Mesh, Route Demands from the main menu. The following dialog will appear displaying an empty log:

Route Demands Dialog, Basic Tab

  1. If you want to execute the algorithm using default routing options, proceed to the next step.  Otherwise, click on the Advanced tab and select the desired advanced routing options.  When complete, click to return to the Basic tab.
  2. Click the Calculate button on the Basic tab to execute the routing algorithm. 

Upon initiation, the Route Demands algorithm will perform the tasks listed below for all mesh demands that are enabled for routing.  

  • Routes the demand working and protect paths over the lightpath layer taking ProtectionType into account.  Note that demands whose FixedLightPathRouting property is set to True will not have their working and protect paths changed.  Demands whose DoNotRoute property is set to True will not be routed.  Demands are routed based on lightpath admin costs.
  • Updates the lightpath TrafficWorking and TrafficProtectDedicated properties which are used to calculate the lighpath Capacity property.  The TrafficProtectShared property is set equal to zero for all lightpaths.
  • Updates the ConduitDiverse and NodeDiverse properties.
  • Lists any UnRouted and NonDiverse Demands in the Results window.
  • Loads network statistics information into the log.

The appearance of network information in the log portion of the dialog is your signal that the routing process is complete.  For tips on evaluating routing results, read the Evaluating Routing Results section of this help file.

Advanced Routing Options

The Advanced tab of the Routing Optimization dialog is shown below.  By making selections in this dialog, you determine which factors take precedence during the routing process.  Your selections will also influence the amount of time required for the algorithm to complete it's analysis.  Use the description of each option (included below) to determine the desired settings.  When your selections are complete, return to the Basic tab and click on the Calculate button to initiate the routing process.

Route Demands Dialog, Advanced Tab

A description of each advanced routing option is provided below.  Unless noted otherwise, any changes you make to option values are retained until the Route Demands dialog is closed. 

Diverse Routing Algorithm

Select one of these three options:

  • Broadleaf:  A diverse routing algorithm based on 's embedded Broadleaf algorithm is used to route demands.
  • Modeler: A diverse routing algorithm, custom developed for the Modeler, is used to route demands. 
  • Both Broadleaf and Modeler:  The Broadleaf algorithm is executed first, followed by the Modeler algorithm in the event that a demand
    • 's routing paths are non-diverse. 

Diversity Iterations

The value of Diversity Iterations is used by the Modeler algorithm.  Diversity Iterations defines the maximum number of attempts to find diverse working and protect paths for each protected demand. The default value is typically sufficient to ensure diversity.  However, certain topologies are more complex and require a greater number of diversity iterations. There are scenarios for which a finding a diverse route is not possible, perhaps due to Conduit Id settings or hard maximum capacity constraints. For these scenarios, you can increase the speed of the algorithm by setting diversity iterations to a lower number.

Diversity Step Size

Diversity Step Size applies to the Modeler algorithm.  The optimum value for step size (optimum being the best result in the least amount of time) can vary greatly depending on the characteristics of a particular network.  To determine a good working value for your network, run the Route Demands algorithm with a range of step size values and observe the impact on routing results.

Defaults

Use the Defaults button to return each selection in the Diverse Routing Options box to it's default value. Default values exist for the Use Broadleaf Algorithm and Use Modeler Algorithm checkboxes, for the number of Diversity Iterations, and for Diversity Step Size. 

Note:  Diverse Routing Option settings are retained indefinitely unless you reset them by clicking on the Defaults button.  Diverse Routing Options include Routing Algorithm, Diversity Iterations, and Diversity Step Size settings.

Routing Constraints

Routing constraints provide a means of setting priorities for lightpath selection during the routing process.  Based on your constraint selections, penalties are applied to potential solutions that do not meet the selected criteria.  By comparing the combined penalties for each potential solution, the Modeler's routing algorithms can identify the solution that best meets your criteria. To select options for routing constraints, click on the Routing Constraints... button.  For a description of routing constraints, read the Routing Constraints section of this help file.

Note:  Routing Constraint settings are retained as long as the current network file remains open.  If you have made any recent changes to routing constraints (regardless of which routing algorithm was in use), click on the Routing Constraints button to verify that constraint values reflect your current needs. 

Detailed Log

When checked, the log will include various property values used by Modeler's routing algorithms while evaluating each demand's working and protect paths. This information is primarily intended to assist Modeler software developers with debugging algorithms.  This is a resource intensive feature and should remain unchecked for most network design projects.

Randomly Sort Demands

When checked (checked is the default), demands will be sorted in a random fashion prior to the algorithm being run.  When more than one demand in a network is eligible to be routed over the same path, the order in which the routing algorithm processes demands will influence the outcome.  

For assistance with the algorithm options described above,  please contact the design tools team at designtools@sycamorenet.com.
